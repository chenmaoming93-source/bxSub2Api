# 项目开发规约

> 本文档用于记录本项目长期有效的开发约定。后续新增规约时，应按主题追加章节，不随意改写或删除既有规则。

## 1. 文档约定

### 1.1 规则级别

- **必须**：强制遵守；不符合时不得交付。
- **应该**：原则上遵守；确需偏离时，应在任务记录中说明原因。
- **建议**：推荐实践，可根据实际情况选择。

### 1.2 新增规约的格式

新增规约时，应包含以下内容：

1. 适用范围。
2. 强制规则。
3. 命名或目录约定。
4. 验证方式。
5. 必要时提供正确示例与禁止示例。

## 2. 数据库表结构变更 SQL 归档规约

### 2.1 适用范围

本规约适用于所有会改变数据库表结构的 DDL，包括但不限于：

- 新建或删除表。
- 新增、修改或删除字段。
- 新增、修改或删除索引、唯一约束及外键。
- 修改字段类型、长度、默认值或可空性。

### 2.2 存放位置

- 表结构变更 SQL **必须**存放在 `backend/sqlArchiving/`。
- **不得**为此类变更在 `backend/migrations/` 中新增 SQL 文件。
- **不得**修改 `backend/migrations/` 中已有的初始化或迁移 SQL 文件来承载新表结构变更。
- `backend/sqlArchiving/` 中的文件仅作为部署、升级和人工执行的 SQL 归档，不参与项目运行时的自动初始化流程。
- 应用代码需要使用新增字段时，仍应同步更新 Ent schema、领域模型、Repository、DTO 等代码；本规约仅改变 DDL SQL 的存放和执行方式。

### 2.3 文件编号与命名

- 文件名格式必须为：`NNN_short_description.sql`。
- `NNN` 必须沿用项目初始化 SQL 的编号序列，并取当前已使用最大编号的下一个编号。
- 编号检查必须同时覆盖：
  - `backend/migrations/*.sql`
  - `backend/sqlArchiving/*.sql`
- 不得复用已有编号。发现同号文件时，应先选择下一个未使用编号。
- 描述部分使用小写英文和下划线，准确表达本次结构变化。

示例：

```text
backend/sqlArchiving/160_api_key_platform_purpose.sql
backend/sqlArchiving/161_add_example_column.sql
```

### 2.4 SQL 内容要求

- SQL **必须**使用项目当前数据库支持的语法，并能够作为独立文件直接执行。
- 文件中不得包含伪代码、模板占位符或需要人工补全的变量。
- 每条 SQL 语句必须以分号结尾。
- 同一个文件应只包含一个内聚的表结构变更主题。
- 应优先使用具备重复执行保护的语法，例如 `IF EXISTS`、`IF NOT EXISTS`；数据库不支持时，应在文件注释中明确执行前置条件。
- 新增非空字段时，必须明确考虑现有数据，并提供可执行的默认值、回填步骤或分阶段约束方案。
- SQL 注释应说明变更目的，不应代替必要的 SQL 语句。
- 不得依赖项目运行时迁移器对 SQL 进行改写后才能执行。

正确示例：

```sql
ALTER TABLE api_keys
    ADD COLUMN IF NOT EXISTS platform VARCHAR(50) NULL,
    ADD COLUMN IF NOT EXISTS purpose VARCHAR(20) NOT NULL DEFAULT 'user_created';
```

禁止示例：

```sql
-- 请部署时自行填写字段类型
ALTER TABLE api_keys ADD COLUMN purpose <TYPE>;
```

### 2.5 生成与验证流程

每次产生表结构变化时，必须依次完成：

1. 检查两个 SQL 目录中的已有编号，确定下一个可用编号。
2. 在 `backend/sqlArchiving/` 新建 SQL 文件。
3. 检查 SQL 方言、字段定义、默认值以及现有数据兼容性。
4. 确认文件可脱离应用程序和运行时迁移器直接执行。
5. 更新相关任务或 MVP 文档中的 SQL 路径与验证证据。

### 2.6 当前编号基线

- 当前已归档文件：`backend/sqlArchiving/160_api_key_platform_purpose.sql`。
- 在没有新增更高编号文件的前提下，下一份 SQL 应从 `161` 开始。
- 实际生成新文件前仍必须重新扫描目录，不得只依赖本文记录的编号。

## 3. 配置文件更新规约

### 3.1 适用范围

本规约适用于新增、删除、重命名或修改后端运行配置项及其默认值、示例值和说明。

### 3.2 强制规则

- 后端配置发生变化时，**必须**同步更新实际开发配置文件 `backend/config/config.yaml`。
- 不得只更新 `deploy/config.example.yaml`、Viper 默认值或配置结构体而遗漏 `backend/config/config.yaml`。
- `backend/config/config.yaml` 中的配置层级、字段名和数据类型必须与后端配置结构及 `mapstructure` 标签一致。
- 涉及敏感配置时，只能写入适合开发环境的安全占位值，不得提交真实密钥、Token、密码或生产连接信息。

### 3.3 验证方式

配置变更交付前必须确认：

1. `backend/config/config.yaml` 包含新增或调整后的配置项。
2. 配置能够被 Viper/mapstructure 正确加载。
3. 配置默认值、开发配置和示例配置之间不存在非预期差异。
4. 相关配置校验与测试通过。
