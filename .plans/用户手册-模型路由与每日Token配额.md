## 1. 功能概述

本次更新为 Sub2API 管理后台新增了三项与模型路由和 Token 用量管控相关的功能：

| 功能 | 界面位置 | 作用 |
|---|---|---|
| **分组模型路由** | 管理后台 → 分组管理 → 创建/编辑分组 | 配置用户请求的分组模型名 → 实际模型 → 账号的映射和优先级 |
| **全局模型每日限额** | 管理后台 → 分组管理 →「模型 Token 限额」按钮 | 全局限制某个实际模型的每日 Token 消耗总量 |
| **用户模型每日限额** | 管理后台 → 用户管理 → 用户操作菜单 →「模型 Token 限额」 | 限制特定用户使用某个实际模型的每日 Token 量 |

三者协同工作的效果是：用户用简化模型名（如 `fast-code`）请求 API 时，系统按优先级选取实际模型和账号，并在三层限额（候选→全局→用户）均未耗尽时放行；任意一层耗尽则自动降级到下一候选。

## 2. 功能一：分组模型路由配置

### 2.1 界面位置

**路径：** 管理后台 → 分组管理 → 点击某个分组的「编辑」按钮（或点击「创建分组」）→ 在弹出窗口中找到「**模型路由配置**」区域。

> 你需要在分组详情弹窗中向下滚动，在「平台」「费率倍数」等基础字段下方可以看到「模型路由配置」卡片。

### 2.2 如何使用

#### 2.2.1 启用模型路由

在「模型路由配置」区域顶部有一个开关，显示**「已禁用 / 已启用」**。点击开启后，才会出现路由规则编辑界面。

#### 2.2.2 添加一条路由规则

一条路由规则定义了一个**路由别名**（用户请求中使用的模型名）到多个**候选实际模型**的映射。

**步骤：**

1. 点击「**添加路由规则**」按钮。
2. 在 **「路由别名」** 输入框中填写一个简洁的名称，例如 `coding-fast`、`chat-pro`。用户将在 API 请求的 `model` 字段中使用此别名。
3. 在候选区域点击「**添加候选**」，为当前别名添加一个候选模型：
   - **上游模型**：填实际模型名，如 `claude-sonnet-4-5`、`gpt-5.2-codex`。这是后端真正请求的模型。
   - **优先级**：填一个非负整数（如 `1`、`2`、`3`）。数字越小优先级越高，系统会先尝试优先级最高的候选。
   - **每日 Token 限额**（可选）：该候选模型在此分组下的每日 Token 上限。留空表示不限。
   - **账号搜索**：在搜索框中输入账号名称关键词，从下拉列表中选择分配给此候选的账号。可多选，已选账号以蓝色标签展示，点击右侧 `×` 可移除。
4. 重复步骤 3 添加更多候选（按优先级从高到低排列）。
5. 重复步骤 1-4 添加更多路由别名。

#### 2.2.3 旧格式兼容

如果你之前使用过旧版 `model_routing`（直接 `{ "claude-opus-*": [1,2,3] }` 格式），编辑分组时系统会自动读取并兼容展示。但**保存后会统一转换为新格式**（含路由别名和候选列表）。旧格式的表单功能不受影响。

#### 2.2.4 校验规则

保存前系统会校验：

- 路由别名不能为空、不能重复
- 每个别名下至少有一个候选
- 每个候选的模型不能为空、同一别名下候选模型不能重复
- 每个候选至少选择一个账号
- 优先级必须是非负整数，同一别名下不能重复
- 每日 Token 限额如有填写，必须是非负整数

#### 2.2.5 保存与生效

填写完成后，点击弹窗底部的「**保存**」按钮。保存成功后，分组配置立即生效。

### 2.3 功能效果

- 用户使用路由别名（如 `fast-code`）作为请求 `model` 字段时，系统按优先级顺序尝试候选实际模型和账号。
- 支持**候选内账号故障转移**：同一候选的多个账号依次尝试。
- 支持**候选间故障转移**：当前候选的所有账号都不可用，或候选/全局/用户三层 Token 限额任一耗尽时，自动尝试下一个候选。
- 全部候选失败后，返回 429 或 503 响应。

## 3. 功能二：全局模型每日 Token 限额

### 3.1 界面位置

**路径：** 管理后台 → 分组管理 → 页面顶部工具栏中的「**模型 Token 限额**」按钮（带图表图标）。

> 该按钮在分组列表页面的顶栏，与「创建分组」「刷新」等按钮并列。

### 3.2 如何使用

1. 点击「**模型 Token 限额**」按钮，弹出「全局模型每日 Token 限额」窗口。
2. 窗口加载后显示已有的全局限额配置，每行包含：
   - **上游模型**：实际模型名
   - **每日 Token 限额**：该模型每天允许的总 Token 数
   - **今日已用**：当前累计已使用量（只读）
3. **添加模型**：点击底部「**添加模型**」，新增一行。在输入框中填写模型名和限额值（留空为不限）。
4. **修改限额**：直接编辑对应行的模型名或限额输入框。
5. **保存**：点击右下角「**保存**」按钮。系统会逐条提交并刷新回显。

### 3.3 功能效果

- 配置后，**所有用户所有分组**对该模型的请求总和受到此限额约束。
- 当日累计 Token 达到限额后，后续所有使用该模型的候选将被跳过，自动降级到下一候选。
- 次日零点（按服务器时区）自动清零，重新计。

### 3.4 校验规则

- 模型名不能为空、不能重复
- 限额值必须留空（不限）或为非负整数

## 4. 功能三：用户模型每日 Token 限额

### 4.1 界面位置

**路径：** 管理后台 → 用户管理 → 在用户列表中找到目标用户 → 点击行尾的「`…`」操作菜单 → 选择「**模型 Token 限额**」。

> 操作菜单中包含「编辑」「API 密钥」「平台额度」「模型 Token 限额」等选项。

### 4.2 如何使用

1. 在用户操作菜单中点击「**模型 Token 限额**」，弹出「用户模型每日 Token 限额」窗口。
2. 窗口标题下方显示当前操作用户的邮箱，以确认操作对象。
3. 每行配额包含：
   - **上游模型**：实际模型名
   - **每日 Token 限额**：该用户每天使用此模型的最大 Token 数
   - **今日已用**：当前累计已使用量（只读）
   - **删除按钮**（垃圾桶图标）：移除该行配额
4. **添加模型**：点击「**添加模型**」，新增一行后填写模型名和限额。
5. **删除模型**：点击行尾垃圾桶图标删除该行配额配置。
6. **保存**：点击右下角「**保存**」按钮，多条配额批量提交并刷新。

### 4.3 功能效果

- 配置后，**仅该用户**对该模型的请求受限，其他用户不受影响。
- 该用户当日该模型 Token 累计达到限额后，后续候选跳过，尝试降级。
- 每日零点自动重置。

### 4.4 校验规则

- 同一用户下模型名不能重复
- 限额值必须留空（不限）或为非负整数

## 5. 后端运行机制（管理员参考）

### 5.1 Token 口径

所有 Token 限额均使用 `input_tokens + output_tokens + cache_creation_tokens + cache_read_tokens` 的总和进行计量。`CountTokens`（仅计数不计费）不计入每日限额。

### 5.2 三层限额优先级

请求到达时，系统依次检查：

1. **候选模型每日限额**（分组级）：该候选配置的 `daily_token_limit`
2. **全局模型每日限额**：`model_token_daily_usages` 表中该模型的全局限额
3. **用户模型每日限额**：`user_model_token_daily_usages` 表中该用户+模型的限额

任一限额耗尽（当日 `used_tokens >= daily_limit_tokens`），当前候选被跳过。

### 5.3 每日重置

每日窗口按服务端全局时区的零点（`timezone.StartOfDay`）切换。跨日后上一日的 `used_tokens` 不影响新一日。

### 5.4 数据存储

- `model_routing` 字段存储在 `groups` 表的 JSON 列中
- `model_token_daily_usages`：全局模型日用量表
- `user_model_token_daily_usages`：用户模型日用量表
- `group_candidate_token_daily_usages`：分组候选模型日用量表
- 使用唯一索引保证每日每维度仅一行，通过数据库原子 `INSERT ... ON DUPLICATE KEY UPDATE` 保证并发不丢增量

### 5.5 计费与日志

每次成功的 API 请求在 `usage_logs` 中记录：

- `model`：用户请求的分组模型名（路由别名）
- `requested_model`：同上，用于匹配计费策略
- `upstream_model`：实际调用的上游模型名，用于限额维度统计

## 6. 常见问题

### Q1：如何让某个分组完全不使用模型路由？

在「模型路由配置」中将开关保持为「**已禁用**」状态。此分组将沿用原有的账号选择逻辑（不考虑路由别名），或者不启用模型路由。

### Q2：全局限额和用户限额同时配置了，以谁为准？

**以先耗尽者为准。** 系统同时检查三个维度，任意一个耗尽即跳过。例如：全局模型限额为 100 万，用户限额为 50 万，用户当天用到 50 万后，该用户的请求就会跳过该候选。

### Q3：限额值设为 0 和不填有什么区别？

**没有区别。** 值 `0`、`null` 和留空均表示「不限制」。系统在判断时会检查 `daily_limit_tokens IS NOT NULL AND daily_limit_tokens > 0`。

### Q4：旧版 model_routing 还能用吗？

**可以。** 系统完全兼容旧格式 `{ "claude-opus-*": [1,2,3] }`。但如果你通过新 UI 编辑并保存，配置会自动转换为新格式（路由别名 + 候选列表）。旧格式的路由行为不受影响。

### Q5：如何查看某个模型当天的总用量？

目前通过管理后台的「全局模型每日 Token 限额」弹窗可以看到每个模型的「**今日已用**」数量。用户维度的今日用量在「用户模型每日 Token 限额」弹窗中查看。

### Q6：配置了路由后，用户的请求 model 写什么？

用户在 API 请求的 `model` 字段中填写**路由别名**（例如 `fast-code`），而不是实际模型名。系统会自动将请求转发到对应的上游模型。

### Q7：候选之间如何 failover？

按优先级数字从小到大依次尝试。如果当前候选的：
- 所有账号都不可调度/限流/报错，**或**
- 三层 Token 限额任一耗尽

则自动跳到下一个优先级的候选。全部候选都失败后返回错误。

### Q8：修改限额后多久生效？

立即生效。配额配置保存后通过缓存失效机制让下一次请求读取到新值。
