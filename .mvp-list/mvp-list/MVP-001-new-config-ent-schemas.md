# MVP-001: Create 3 new config Ent schemas

- Protocol: `mvp-list/v1`
- State: `VERIFIED`
- Estimate: `~20 min`
- Estimate rationale: Three small, structurally identical ent schema files. Each ~45 lines, following existing patterns in `model_token_daily_usage.go`. Boilerplate work with clear templates.
- Dependencies: none

## Outcome

Three new Ent schema files exist for the config tables, matching the migration 158 SQL structure and ready for `go generate ./ent`.

## Context

Migration `158_token_quota_config_and_usage_split.sql` defines three new config tables to separate daily limit configuration from daily usage recording. The corresponding Ent schemas need to be created so Ent can generate type-safe Go code for CRUD operations.

Reference patterns exist in:
- `backend/ent/schema/model_token_daily_usage.go` — field definition, mixin, index, annotation patterns
- `backend/ent/schema/user_model_token_daily_usage.go` — edge definition pattern (FK to users)
- `backend/ent/schema/group_candidate_token_daily_usage.go` — edge definition pattern (FK to groups)

## In Scope

- `backend/ent/schema/model_token_daily_limit_config.go` — model-level config with `UNIQUE (model)`
- `backend/ent/schema/user_model_token_daily_limit_config.go` — user+model config with FK to `users`, `UNIQUE (user_id, model)`
- `backend/ent/schema/group_candidate_token_daily_limit_config.go` — group+route_alias+upstream_model config with FK to `groups`, `UNIQUE (group_id, route_alias, upstream_model)`

Each schema must include:
- `mixins.TimeMixin{}` (provides `created_at`, `updated_at`)
- `entsql.Annotation{Table: "..."}` matching the migration SQL table name
- `id` field (auto-generated by ent via `int64` with default)
- `daily_limit_tokens` as `field.Int64(...).Optional().Nillable()`
- Correct unique indexes
- Edges for FK relationships (user_model → User, group_candidate → Group)

## Out of Scope

- Modifying existing usage schemas (MVP-002)
- Running `go generate ./ent` (MVP-002)
- Any repository or service changes

## Implementation Notes

- Study `model_token_daily_usage.go` for annotation + mixin + index patterns
- Study `user_model_token_daily_usage.go` for the `edge.From("user", User.Type)` pattern; the config schema should have `edge.To` or check if User schema has a backref
- Study `group_candidate_token_daily_usage.go` for the group FK edge pattern
- The `id` field uses ent's implicit ID (just declare `field.Int64("id").Immutable()` or rely on ent's built-in ID); check existing patterns — `mixins.TimeMixin` + explicit ID is the convention in this codebase
- `nonNegative` validator from usage schemas should be reused for `daily_limit_tokens`
